The first article about RLMeta has this example grammar for a calculator:

Calculator {
  expression =
    | additive
  additive =
    | multitive:x '+' additive:y -> add(x y)
    | multitive
  multitive =
    | digit:x '*' multitive:y    -> mul(x y)
    | digit
  digit =
    | '0'-'9':x                  -> int(x)
}

At a first glance, it looks correct, but it has a problem with operator associativity. It is avoided by only supporting operators that evaluate the same no matter if evaluated left or right.

In this article I show how to correctly parse any binary operator expression in RLMeta.

What is operator associativity?

Operator associativity has to do with the order that operators are evaluated. The following expression can be evaluated in two ways:

1-2-3

Either as this (if the operator is left associative):

(1-2)-3

Or as this (if the operator is right associative):

1-(2-3)

Let's see how the calculator evaluates expressions.

Right

The following grammar works like the calculator but only supports the subtraction operator:

  1. right.rlmeta
Calculator {
  expr  = digit:x '-' expr:y -> sub(x y)
        | digit
  digit = '0'-'9':x          -> int(x)
}

The following support functions either create an AST node or evaluates the expression:

  1. ast.py
def sub(left, right):
    return ["-", left, right]
  1. eval.py
def sub(left, right):
    return left - right

For an expression with only two numbers, the grammar works as expected. But with more numbers, the expression created is incorrect.

The calculator evaluates 1-2-3 as 1-(2-3) which is incorrect because subtraction is left associative. So is addition, but evaluating addition according to a right associative parse gives the same result.

If we strip the semantic actions from the expr rule it is easier to see what creates this right associative parse:

expr = digit '-' expr | digit

Left

In order to get a left associate parse (that we need for subtraction) we would like to write the expr rule like this instead:

expr = expr '-' digit | digit

The whole grammar would then look like this:

  1. left.rlmeta
Calculator {
  expr  = expr:x '-' digit:y -> sub(x y)
        | digit
  digit = '0'-'9':x          -> int(x)
}

But this will not work with the parsing algorithm that RLMeta uses. In order to parse an expr it first has to parse an expr and so on, and it will get stuck in an infinite loop.

Left correct

The right associative parse:

expr = digit '-' expr | digit

Can be expressed with a repetition instead:

expr = digit ('-' digit)*

This parses the same expressions. How to construct a left associate parse from this? The whole grammar looks like this:

  1. left_correct.rlmeta
Calculator {
  expr  = digit:x (op:y digit:z -> [y z])*:xs -> leftAssoc(x xs)
  digit = '0'-'9':x                           -> int(x)
  op    = '-'                                 -> makeSub()
}

In the semantic action for expr, x is bound to a digit, and xs is bound to a list of pairs consisting of an operator and a digit: [[op, digit], [op, digit], ...]. The function leftAssoc turns this into a left associate tree:

  1. support.py
def leftAssoc(expr, items):
    while items:
        op, rhs = items.pop(0)
        expr = op(expr, rhs)
    return expr

def rightAssoc(expr, items):
    if items:

        expr = op(expr, rightAssoc(rhs, items))

    while items:

        expr = op(expr, rhs)
    return expr

Nicer handling of precedence

  1. calculator_ops.rlmeta
Calculator {
  expr  = digit:x (op:y digit:z -> [y z])*:xs -> parseOps(x xs)
  digit = '0'-'9':x                           -> int(x)
  op    =
    | '**' -> Op(makePow() "3" "right")
    | '*'  -> Op(makeMul() "2" "left")
    | '/'  -> Op(makeDiv() "2" "left")
    | '+'  -> Op(makeAdd() "1" "left")
    | '-'  -> Op(makeSub() "1" "left")
}
  1. support.py
class Op(object):

    def __init__(self, fn, prec, assoc):
        self.fn = fn
        self.prec = int(prec)
        self.assoc = assoc
  1. support.py
def parseOps(expr, items, min_level=1):
    while items and items[0][0].prec >= min_level:
        op, rhs = items.pop(0)
        if op.assoc == "left":
            next_level = min_level + 1
        else:
            next_level = min_level
        expr = op.fn(expr, parseOps(rhs, items, next_level))
    return expr

compile() {
    echo "import sys"
    echo "import pprint"
    rlmeta --support
    rlmeta < "$1"
    cat "support.py"
    cat "$2"
    echo "makeAdd = lambda: add"
    echo "makeSub = lambda: sub"
    echo "makeMul = lambda: mul"
    echo "makeDiv = lambda: div"
    echo "makePow = lambda: pow"
    echo "try:"
    echo "    for expr in sys.stdin.read().splitlines():"
    echo "        pprint.pprint(Calculator().run('expr', expr), width=20)"
    echo "except _MatchError as e:"
    echo "    sys.stderr.write(e.describe())"
}

(
    while read -e expr; do
        echo "$expr" | python <(compile "$1" "ast.py")
        echo "=>"
        echo "$expr" | python <(compile "$1" "eval.py")
        echo ""
    done
